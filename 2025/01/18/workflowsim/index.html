WorkflowSim: A Toolkit for Simulating Scientific Workflows in Distributed Environments

related work

  • disadvantages of CloudSim
    • only supports the execution of single workloads
      • does not consider dependencies or clustering
    • ignore the occurence of failures and overheads
  • task clustering
    • static
    • not consider middleware overhead from diverse grid middelware services
  • failure analysis and modeling present

models and features

workflow management system

  • workflow mapper: map abstract workflows to concrete, dependent on execfution sites
  • workflow engine: handle data dependencies
  • workflow scheduler: match jobs to resources
  • clustering engine: merge small tasks into a large hob
  • provenance collector: tracks the history of task/job execution
  • workflow partitioner: divide user workflow into multiple sub-workflows

components

workflow mapper

model workflows as DAG. mapper is used to import DAG files formatted in XML

cluster engine
  • job vs task
    • task: program tath a user would like to execute
    • job: an atomic unit seen by the exeuction system, contains multiple tasks to be executed in sequence or parallel

cluster engine merge task into jobs

workflow engine

make sure a job is released when all its parent jobs have completed successfully

use DAGMan as the workflow engine

workflow scheduler and job execution
  • static: jobs are assinged to a worker node at workflow panning stage. when a job reach remote scheduler, it will wialt until the assinged worker node is free.
  • dynamic: jobs are matched to a worker node in the remote scheduler whenever a worker node becomes idle.

layered overhead

  • workflow engine delay: 上一个parent job完成(released)到下一个job被提交(into the ready queue)的时间
  • queue delay: 一个job被workflow engine提交到queue中到local scheduler发现这个job已经在running状态
  • poststript delay & prescript delay: job执行前后的脚本执行的时间.
  • data transfer delay: 节点之间传输数据的延时 [stage in, clean up, stage out]
  • clustering delay: 用来衡量真正的运行时间和workflow scheduler认为的运行时间. 因为把job包装起来也需要消耗一定的时间

Layered Failures and Job Retry

  • task failure VS job failure
    • task failure: 一个task失败只影响自己
    • job failure: 一个task失败还影响其他task
  • failure generator: ? 为什么要产生failure?
  • failure monitor: collect failure records, return to workflow management system
  • options to improve workflow perfomance
    • 选择重试整个job,或者只运行failure的部分 -> workflow scheduler里集成,将根据用户的选择执行
    • reclustering: 根据failure rate来调整 clustering strategy -> workflow engine里集成

validation

GPUCloudSim

  • contributions
    • enable cloudsim to use GPU-features
    • models to simulate interference among co-running applications, the overhead of virtualization and power consumption of GPUs.
    • multi-level scheduling and provisioning of GPU resources.
    • VM placement greedy algorithm

proposed GPUCloudSim extention

  • GPUCloudSim layer provides necessary classes to model and simulate GPU-enabled
  • interference-aware GPUCloudSim layer gives GPUCloudSim the ability to model the interfenrece of co-running app in a GPU
  • perfomance-aware GPUCloudSim layer provides its underlying layer with the ability to model the virtualization-incurred performance degradation of virtualized GPUs
  • Power-aware GPUCloudSim layer enables extention to simulate power-aware GPUs

cloud layer

  • SimEntity: abstract class extended by event-aware entities within the simulator
  • CloudSimTags contains the event
  • DatacenterBroker and Datacneter are CloudSim classes that use events to comunicate.
    • DatacenterBroker acts on behalf of end-users; queries data centers fo r allocation of resources and then submits computations. to support GPU workdloads
    • Datacenter represents the hardware infrastructure for a datacenter to support GPU-equipped hosts
  • PowerGpuDatacenter represents a power- and engery-aware GPU-enabled data center.

server layer

  • classes in GPUCloudSim layer: provide basic support for GPU hardware and software in hosts and vms
    • GpuVm: extends CloudSim’s Vm class to model a VM with a vGPU attached
    • Vgpu: represents a vGPU that is attached to a VM
    • GpuHost: extends Host class from CloudSim to model a GPU-capable host with one ore more video cards attached
    • VgpuScheduler: abstract class that defines the policy to share the processing power of a video card among resident vGPUs
    • Pgpu: the Pgpu class represents a physical GPU. A Pgup incorporates multiple processing elements that represent GPU SMs.
    • VideoCard: represents a video card in the system. a video card has one ore more Pgpu on board, each with its own memory
    • VideoCardAllocationPolicy: abstract class represents the provisioning policy of video cards to vGPUs. each GpuHost has its own video card allocation policy
    • VideoCardBwProvisioner: abstract class, represents the provisioning policy used by a video card to allocated PCIe bandwidth for its GPUs. each video card has its own PCIe provisioner
    • PgpuSelectionPolicy: an interface implemented by classes that provide a GPU selection policy for vGPUs on a video card. when there are multiple available GPUs for a vGPU on a video card, the selection policy determines which GPU to select.
  • classes in performance-aware GPUCloudSim layer.
    • PerformanceGpuHost: extends GpuHost to support schedulers that model the virtualization overhead for vGPUs. implements both performanceScheduler and performanceModel interfaces
    • PerformanceScheduler: an interface implemented by a VgpuScheduler with a performance degradation model, in order to become compatible with PerformanceGpuHost
    • PerformanceModel: interface implemented by a VgpuScheduler to model virtulization overheads for vGPUs
  • classes in power-aware GPUCloudSim layer
    • PowerGpuHost: extends PerformanceGpuHost to support power-aware video cards and incorporate a power model for other host components. a power model for the host needs to implement the PowerModel interface, provided by the power package of CloudSim
    • PowerVideoCard: represents a power-aware video card that incorporates a video card model
    • VideoCardPowerModel: interface needs to be implemented by the classes that provide a power model for video cards
  • implemented scheduling and provisioning policy:
    • VpuSchedulerTimeShared
    • VgpuSchedulerSpaceShared
    • VgpuSchedulerFairShare

gpu layer

  • classes in GPUCloudSim layer
    • Pe: a CloudSim class that represents a processing elemnt. used to simulate an Sm in the GPU
    • PeProvisioner: abstract class from CloudSim represents the provisioning policy that is used to allocatate the processing power of a Pe to vGPUs
    • GpuBwProvisioner: abstract class that represents the provisioning policy of a GPU to allocate GPU memory bandwidth for resident vGPUs
    • GpuGddramProvisioner: abstract class represents the provisioning policy used by a GPU to allocate GPU memroy for resident vGPUs
    • GpuCloudlet: extends Cloudlet class from CloudSim to represent an application with both CPU and GPU portions of execution
    • GpuTask: represents a computation that is executed on the GPU. includes the number of thread blocks and the length of each block. in addition, it incorporates utilization models to imitate resource demands
    • MemoryTransfer: internal class that represents a memory transfer that occurs within a GpuTask context
    • GpuTaskScheduler: abstract class that represents the scheduling policy employed by a Vgpu to execute its GPu tasks.
      • GpuTaskSchedulerLeftover extends above to implement the cooperative multitasking scheduler based on leftover policy/
  • classes in interface-aware GpuCloudSim layer, model inter-application interference that may slow down co-running GPU tasks
    • InterferenceModel: interface needs to be implemented in order to provide a model for inter-application interference
    • InterferenceGpuTaskSchedulerLeftover: extends the GpuTaskSchedulerLeftover class to add supoort for interference models. co-running GPu tasks may experience a slow down depending on their utilization of shared resources.
